# Сервер для кофейни

## Постановка задачи

Сегодня тебе предстоит разработать сервер для приложения кофейни.

Фронтендеры уже написали интерфейс для приложения и ждут, пока будет готов сервер.

Приложение позволяет просматривать список доступных напитков и их цену. В приложении три экрана.

**Главный экран.** На нем выводится список всех доступных напитков.

**Экран информации.** На этом экране выводится подробная информация о напитке.

**Доступные напитки.** Данный экран выводит список только из напитков, которые есть в наличии.

Также в приложении есть часть админа, из которой можно добавлять, удалять и изменять напитки.

Фронтендеры отправили нам небольшое техническое задание, в котором описаны ожидания от нашего сервера.

## Техническое задание

### Эндпоинты

На сервере должны быть реализованы следующие [эндпоинты](https://wp-kama.ru/handbook/rest/basic#routes-endpoints):

`GET /drinks` – список всех напитков  
`GET /drinks/in-stock` – список всех напитков, которые есть в наличии  
`GET /drinks/:id` – подробная информация о конкретном напитке  
`POST /drinks` – добавление нового напитка  
`DELETE /drinks/:id` – удаление конкретного напитка  
`PATCH /drinks/:id` – изменение конкретного напитка

Каждый напиток должен содержать следующую информацию:

- Название
- Цена
- Есть ли напиток в наличии
- Содержит ли напиток кофеин
- Объем
- Описание напитка

При запросе списка напитков необходимо вывести только поля `идентификатор`, `название`, `цена`, то есть формат ответа должен быть примерно таким:

```json
[
  {
    "id": 1,
    "name": "Кофе",
    "price": 100
  }
  {
    "id": 2,
    "name": "Чай",
    "price": 50
  }
]
```

При запросе одного конкретного напитка нужно вывести всю информацию, например:

```json
{
  "id": 1,
  "name": "Кофе",
  "price": 100,
  "inStock": true,
  "containsCaffeine": true,
  "volume": 100,
  "description": "Описание напитка"
}
```

### Обработка ошибок

Если вдруг какая-то операция на сервере провалилась он должен вернуть JSON-объект, содержащий ключ `error` с описанием ошибки. Например:

```json
{ "error": "Ошибка при добавлении нового напитка" }
```

## Releases

### Release 0: проектирование архитектуры

В данном релизе нет кода. Тебе просто нужно подумать о том, каким будет структура сервера.

Подумай над тем, какие нужны структуры, роуты, функции и т.д.

Какие поля должна содержать структура, чтобы она отвечала требованиям технического задания?

Мысленно пройдись по всему будущему проекту – от файла `main.go` до файла структур и сервисов (функций). Что каждый из них должен содержать?

Когда тебе покажется, что ты предусмотрел все аспекты приступай к следующему релизу.

### Релиз 1: скелет сервера

Создай новый проект go и установи gin. Пропиши весь начальный код в файле `main.go`, например функцию main, импорт и запуск сервера на gin.

Инициализируй git и сделай первый коммит.

### Релиз 2: структура напитка

Как и всегда для работы с какой-либо сущностью мы сначала создаём её структуру, чтобы описать какие поля (то есть свойства или аттрибуты) эта сущность может иметь.

Выше мы уже получили техзадание от нелюбимых нами фронтендеров, по которому можно понять какие поля должны быть у структуры напитка. Самой прямой наводкой на названия и количество полей является пример ответа JSON-сервера, который приведен выше.

В данном релизе подготовь эту структуру, попроси тимлида проверить правильность. Если всё ок - делай коммит и двигайся к следующему релизу.

### Релиз 3: простое хранилище в памяти

Нам нужна имитация базы данных, чтобы где-то хранить список напитков и работать с ними.  
**На этом этапе мы не используем файловую систему и БД** — все данные будут жить в памяти на время работы сервера как обычные переменные.

Возможный вариант:

```go
var drinks []Drink
```

Подумай как должна выглядеть структура `Drink`, какие у нее должны быть поля и с какими типами. Для этого ориентируйся на JSON, который приведен в начале задания.

Дальше подготовь функции для работы с этим хранилищем (в отдельном пакете `storage`/`repository` или прямо рядом с `main`):

- получить список всех напитков;
- найти напиток по `id`;
- отфильтровать только напитки «в наличии»;
- создать новый напиток с новым `id`;
- обновить существующий напиток;
- удалить напиток по `id`.

Хендлеры Gin будут вызывать эти функции и не знать, **как именно** ты хранишь данные.

### Релиз 4: роуты и сервисы

У нас всё готово для того, чтобы начать реализацию роутов из техзадания.

Начни с самого первого эндпоинта из задания - `GET /drinks`. Пропиши для него роут в `main.go` и функцию обработки запроса. Функция обработки запроса будет обращаться к сервису, который будет предоставлять нам данные. Следовательно, нужно подготовить и сам сервис, чтобы наш роут начал правильно работать и выдавать клиенту нужный ответ.

Незавершенный пример создания роута и хендлера:

```go
// ...какой-то код...

r.GET("/drinks", func(c *gin.Context) {
  drinks, err := service.GetDrinks()

  // обработка ошибки err

  c.JSON(http.StatusOK, drinks)
});

// ...какой-то код...
```

Мы могли бы весь код из `service.GetDrinks()` написать прямо внутри нашей функции-обработчика, но тогда код станет нечитабельным и его сложно будет сопровождать в будущем. Именно поэтому мы часть логики выносим в `service.GetDrinks()`, в обработчике оставляем только логику ответа клиенту, т.е определяем выдавать ему ошибку или данные и в каком виде это делать.

Когда эндпоинт будет готов протестируй его в Postman. Если всё работает, то сделай коммит и переходи к реализации следующего роута.
